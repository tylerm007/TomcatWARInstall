Key Do's and Don'ts

You will probably pick up API Creation quite quickly, since it's a point and click technique for building a commonly understood API. {{productSlimName}}'s most unique feature is Reactive Programming for business logic, so it's important to be aware of that.

Don't

Do
Code Logic in client Specify Logic in Server to:
  • Centralizes your logic for re-use, including for non-UI clients such as other systems.
  • Improves performance by reducing the amount of data which must be sent to the client
Code Logic in Events Think Spreadsheet - Spreadsheet-like rules are orders of magnitude more expressive than events

Event-driven programming is a common programming model, and fully supported so there are no boundaries on what you can address.

But it's a big mistake to use it instead of Reactive Programming spreadsheet-like rules. Reactive automates Watch/React services, which otherwise require significant code.

For example, it is a common error to do sum/count rollups in event handlers - sum/count rules are much easier, and are apt to perform better.

API Enforces Business Policy


Per Enterprise-class requirements, Business Policies are enforced by the API.

To drive quality and simplify compliance verification, policies are specified at the table level, and automatically re-used over all Custom Resources defined on the underlying table.



Logic is enforced with a combination of server-side JavaScript and Rules, based on a common underlying Object Model.

Security is enforced down to the row and column level, based on roles. Authentication integrates with corporate user lists such as Active Directory or oAuth, based on a pluggable Authentication Manager.


Automatic Object Model and Logic Enforcement


The key concept underlying logic enforcement is the server-side JavaScript Object Model. It's created automatically when you connect, and consists of a row type for each table in your schema (e.g., row types for Customers, Orders, Items, etc.). To sync subsequennt schema changes, click Reload Schema.

You access this same underlying object model in rules and events, using system-supplied row/oldRow variables. These provide:
  • ORM Services:
    • Access to attributes and related objects, and
    • Persistence services to read, write and create rows, within a system-maintained Transaction Cache
  • Logic Execution Services to invoke the rules and JavaScript events in response to client updates (summarized below)




Declarative Reactive Rules are JavaScript functions that are attached to columns to derive their values, and to tables to validate updates. Rules differ from procedural code in key ways:

  • Automatic Invocation: the {{productAPIServer}} applies the rules that are relevant to each incoming request. You don't call the rules, so a key source of error is eliminated.
  • Automatic Ordering: the {{productAPIServer}} executes the rules in an order that reflects their dependencies. You don't order your logic, so a key source of maintenance effort is eliminated.
  • Chaining: rules can depend on the results of other rules, so 1 change might trigger another - even in another table, automatic multi-table transactions.
  • Automatic Persistence: as rules chain across tables, SQL commands are automated to read / write (and cache) related data, and bracket transactions. SQL is optimized to eliminate and reduce database traffic.
 The diagram illustrates system logic execution in response to client updates (Puts, Posts, Deletes) - a set of rows of various types (orders, items):

  • For each submitted row, the system
    • de-aliases it to create row objects, so logic is automatically shared over all Resources
    • performs optimistic locking checks, and obtains the oldRow  (for state transition logic)
    • invokes your rules and events

  • Rules and events changes can propagate, that is, they may change data referenced by other rules and cause them to fire ("forward chaining")
    • Multi-table: such rules can be in other tables.
      • For example, a line item quantity will adjust the amount, which adjusts the orders' amount_total, which adjusts the customers' balance.
    • All SQL is transparent and automated

  • If no errors are detected, the transaction is committed. Otherwise, an exception is returned to the client
